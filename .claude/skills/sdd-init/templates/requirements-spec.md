# 需求规格编写规范

> 适用范围：所有通过 SDD 框架管理的变更
> 文档用途：指导 AI 和团队成员编写符合标准的需求规格

## 文档结构

每份 `spec.md` 必须包含以下章节：

```
1. 概述
2. 功能需求（FR）
3. 非功能需求（NFR）
4. 业务规则（BR）
5. 数据模型
6. API 接口（如有变更）
7. 验收标准
```

## 编号规则

| 类型 | 前缀 | 示例 | 说明 |
|------|------|------|------|
| 功能需求 | FR- | FR-001 | 系统必须提供的功能 |
| 非功能需求 | NFR- | NFR-001 | 性能、安全、可用性等 |
| 业务规则 | BR- | BR-001 | 业务逻辑约束 |
| 测试用例 | TC- | TC-F001 | 对应 FR 的测试 |

每个变更内编号从 001 开始，不跨变更复用。

## 功能需求（FR）写法

```markdown
### FR-001：<功能标题>

**描述**：系统应能/必须 <做什么>，当 <触发条件> 时，以便 <业务价值>。

**触发条件**：
- <条件1>

**主要流程**：
1. <步骤1>
2. <步骤2>

**异常流程**：
- 当 <异常情况> 时，系统应 <响应>

**相关原型**：prototypes.html #<section-id>
```

写作原则：
- 描述"做什么"，不描述"怎么做"（What, not How）
- 使用"系统应"（should）或"系统必须"（must）区分优先级
- 每条 FR 对应一个可测试的行为
- 避免模糊词汇："快速"→ 具体为"响应时间 < 2秒"

## 非功能需求（NFR）写法

```markdown
### NFR-001：<性能/安全/可用性等类型>

**要求**：<具体、可测量的指标>
**测量方法**：<如何验证达标>
**优先级**：必须 / 应该 / 可选
```

常见 NFR 类型：
- **性能**：响应时间、吞吐量、并发用户数
- **安全**：认证方式、数据加密、权限控制
- **可用性**：系统可用率（如 99.9%）
- **兼容性**：浏览器/设备/OS 版本要求
- **可维护性**：日志要求、监控要求

## 业务规则（BR）写法

```markdown
### BR-001：<规则标题>

**规则**：<明确的业务约束条件>
**示例**：<举例说明规则的应用场景>
**违反后果**：<系统如何处理违规情况>
```

## 验收标准写法

每条验收标准对应至少一个 FR：

```markdown
## 验收标准

| 编号 | 描述 | 验证方式 | 对应需求 |
|------|------|----------|----------|
| AC-001 | <具体可验证的标准> | 手工测试/自动化 | FR-001 |
```

验收标准必须满足 SMART 原则：
- **Specific**：具体明确
- **Measurable**：可量化
- **Achievable**：可实现
- **Relevant**：与需求相关
- **Testable**：可测试验证

## 常见错误清单

❌ 避免以下写法：
- "系统应该友好" → 改为"错误提示应在输入框下方显示，文字 < 20 字"
- "处理速度要快" → 改为"列表加载时间 < 1.5 秒（网络正常条件下）"
- "支持多种格式" → 改为"支持 PDF、Word（.docx）、Excel（.xlsx）格式"
- 多个需求写在一条 FR 里 → 拆分为独立 FR
