# 技术架构规范

> 适用范围：SDD 框架内所有技术实施变更
> 目的：确保架构决策的一致性，避免在提案和实施中引入冲突的架构模式

## 架构评估清单

在 `/sdd:propose` 阶段，对每次涉及技术变更的提案必须回答以下问题：

### 前端架构
- [ ] 组件是否遵循单一职责原则（每个组件只做一件事）？
- [ ] 状态管理：局部状态 vs 全局状态，是否有明确边界？
- [ ] 数据获取：是否有统一的 API 调用层（不在组件内直接 fetch）？
- [ ] 错误边界：是否有全局错误处理？

### 后端架构
- [ ] API 是否遵循 RESTful 规范（或已有的接口规范）？
- [ ] 业务逻辑是否在服务层（不在路由/控制器内直接写逻辑）？
- [ ] 数据访问：是否通过数据访问层（Repository/DAO 模式）？
- [ ] 认证鉴权：新接口是否接入了现有的认证中间件？

### 数据库
- [ ] 新表/字段是否有对应的迁移脚本？
- [ ] 是否评估了索引需求（查询字段是否建索引）？
- [ ] 是否考虑了数据量增长对查询性能的影响？

### 安全
- [ ] 用户输入是否经过验证和清洗（防 SQL 注入、XSS）？
- [ ] 敏感数据是否加密存储（密码、Token 等）？
- [ ] 新 API 是否经过权限检查？

## 分层架构参考

```
前端
├── pages/          # 页面级组件（路由入口）
├── components/     # 可复用 UI 组件
├── hooks/          # 自定义 Hooks
├── services/       # API 调用封装
├── store/          # 全局状态
└── utils/          # 工具函数

后端
├── routes/         # 路由定义（只做参数解析和调用）
├── controllers/    # 请求处理（轻量）
├── services/       # 业务逻辑（核心层）
├── repositories/   # 数据访问
├── models/         # 数据模型定义
└── middlewares/    # 中间件（认证、日志、错误处理）
```
