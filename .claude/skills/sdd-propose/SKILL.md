# SDD Propose — 阶段2：提案评审

## 触发方式
用户输入 `/sdd:propose`

## 前置检查
- 检查 `openspec/changes/active/` 下是否有进行中的变更
- 读取对应的 `change.md` 确认阶段1已完成
- 如未完成阶段1，提示用户先运行 `/sdd:new`

## 执行步骤

### 1. 读取上下文（查询 wiki）
本步骤等价于对 llmwiki 的一次**查询（Query）**：先用导航定位，再钻入细节，避免盲扫整个目录。
- 读取 `openspec/changes/active/<变更名>/change.md`
- 读取 `llmwiki/index.md` 与 `llmwiki/overview.md`，据此**定位**与本次变更相关的模块规格页
- 钻入 `llmwiki/specs-released/` 下定位到的相关规格（含其"交叉引用"指向的关联模块）
- 读取 `knowledge/rules/` 下相关规范

### 2. 生成原型（prototypes.html）
在变更目录下创建 `prototypes.html`：
- 使用纯 HTML + CSS + 少量内联 JS（无需外部依赖）
- 覆盖所有涉及的页面/交互流程
- 每个页面用 `<section id="page-name">` 分隔
- 配色使用系统色（#007AFF / #34C759 / #FF3B30 等），保持简洁专业

原型必须真实可点击，能演示核心用户流程。

**除可点击界面外，原型必须包含以下三类说明信息：**

#### 2.1 用户任务流程（端到端流程导览页）
在原型最前面增加一个独立的"流程导览"页（`<section id="flow-overview">`，默认首屏展示）：
- 用步骤条 / 泳道图形式，展示完整的端到端用户任务路径：**谁（角色）→ 做什么（动作）→ 到哪个页面（节点）→ 产出什么状态**
- 覆盖本次变更涉及的所有主流程与关键分支（如审批通过/驳回）
- 每个流程节点可点击，跳转到对应的 `<section id="...">` 页面
- 若涉及多角色，按角色分泳道；若涉及状态流转，标注状态变化（如 草稿→审批中→已发布）

#### 2.2 页面功能描述（页内信息面板）
每个页面 `<section>` 顶部固定一个"功能说明"信息面板（可用浅色卡片样式，与界面区分），简述：
- **页面用途**：这个页面是做什么的、解决什么问题
- **主要功能点**：列出该页面提供的核心操作（增删改查、导出、审批等）
- **进入/离开**：上一步从哪个页面/动作进入，操作完成后去向哪个页面

#### 2.3 交互逻辑（页内交互说明）
在功能说明面板内或紧邻的位置，标注该页面的关键交互逻辑：
- **操作触发**：用户点击某按钮/控件后发生什么（弹窗、跳转、状态变更、校验提示）
- **条件显示**：哪些字段/控件在何种条件下才出现或可用（如"审计方式=联合时显示联合信息"）
- **校验与异常**：必填校验、边界限制、错误提示文案与位置
- **状态联动**：不同状态下按钮的可用性差异（如已发布状态隐藏"编辑"按钮）

> 信息面板应与可交互界面在视觉上明确区分（例如统一用 `.info-panel` 浅灰卡片 + "ℹ️ 说明"标题），便于评审者一眼区分"原型界面"与"设计说明"。三类说明须与 spec.md 的 FR/BR 编号相互呼应（在说明中引用对应编号）。

### 3. 生成提案文档（proposal.md）
```markdown
# 提案：<变更名>

## 执行摘要
<2-3句话总结本次变更的核心价值>

## 问题分析
<现状描述 + 痛点>

## 解决方案
<方案概述，引用原型页面>

## 技术影响评估
- 前端改动：<描述>
- 后端改动：<描述>
- 数据库改动：<描述>
- 第三方依赖：<描述>

## 风险与应对
| 风险 | 影响 | 应对策略 |
|------|------|----------|

## 工作量估算
| 模块 | 估时 |
|------|------|
| 合计 | <总估时> |
```

### 4. 生成需求规格（spec.md）
按照 `knowledge/rules/requirements-spec.md` 规范生成详细需求规格：
- 功能需求（FR-001 格式编号）
- 非功能需求（NFR-001 格式编号）
- 业务规则（BR-001 格式编号）
- 数据模型变更
- API 接口变更（如有）
- 验收标准

### 5. 生成任务分解（更新 tasks.md）
基于规格拆分具体开发任务，格式：
```markdown
## 待办任务
- [ ] FE-001 <前端任务描述>（估时：Xh）
- [ ] BE-001 <后端任务描述>（估时：Xh）
- [ ] TEST-001 <测试任务描述>（估时：Xh）
```

### 6. 生成实施计划（plan.md）
```markdown
# 实施计划：<变更名>

## 实施顺序
1. 前端原型实现（严格匹配 prototypes.html）
2. 后端接口实现
3. 前后端联调
4. 测试用例执行
5. 验证与归档

## 关键节点
| 节点 | 目标 | 负责人 |
|------|------|--------|
```

### 7. 请求审批
展示生成的所有文档摘要，询问用户：

> 以上提案已生成。请确认：
> 1. 原型界面是否符合预期？（如需修改，请描述）
> 2. 流程导览页的用户任务流程是否准确完整？
> 3. 各页面的功能描述与交互逻辑说明是否清晰无误？
> 4. 需求规格是否完整？
> 5. 批准后将进入实施阶段。
>
> 输入 **批准** 继续，或描述需要修改的内容。

### 8. 审批通过后
将 `openspec/changes/active/<变更名>/` 目录内容复制到 `openspec/changes/approved/<变更名>/`。

更新 `change.md` 阶段状态：
```
- [x] 阶段2：提案评审 — <YYYY-MM-DD>（已批准）
```

告知用户：
- 提案已批准，文件已复制到 `approved/`
- 下一步：运行 `/sdd:apply-change` 进入阶段3（实施变更）
