# SDD Wiki Lint — llmwiki 健康体检

## 触发方式
用户输入 `/sdd:wiki-lint`

## 用途
按需对 llmwiki 做健康检查，保持这个"持续累积的产物"不随规模增长而腐化。
不属于6阶段变更流程，可在任意时刻独立运行（典型时机：几次归档之后、开始大型提案之前）。

## 设计原则
- **只读为主，改动需确认**：体检默认只诊断、出清单；修复操作逐项向用户确认后再执行。
- **不臆造内容**：发现"缺失"只报告并建议，不擅自编造规格内容填补。

## 执行步骤

### 1. 加载 wiki 全貌
- 读取 `llmwiki/index.md`、`llmwiki/overview.md`、`llmwiki/log.md`
- 列出 `llmwiki/specs-released/` 下所有规格页及其 frontmatter

### 2. 逐项检查
对照以下清单逐条核查，记录命中的问题：

| 检查项 | 含义 | 如何发现 |
|--------|------|----------|
| 矛盾 | 两个规格页对同一行为/规则给出冲突描述 | 比对相关模块的 FR/BR |
| 陈旧声明 | 旧规格的描述已被更晚归档的变更推翻，但未更新 | 对照 `log.md` 时间线与各页 `last_updated` |
| 孤立页 | 某规格页既无出链也无入链 | 检查"交叉引用"小节与全库链接 |
| 缺失交叉引用 | A 页引用了 B 的概念却没建链接 | 文本提及模块名/术语但无 `[..](..)` |
| 缺概念页 | 多个规格反复提到某术语/实体，却无 glossary/data-model 条目 | 高频术语统计 |
| 索引失配 | `index.md` 与 `specs-released/` 实际文件不一致（多/少/摘要过时） | 目录比对 |
| frontmatter 缺失/失效 | 规格页缺 frontmatter，或 source_changes 指向不存在的归档 | 校验每页头部 |
| overview 失配 | `overview.md` 模块全景与实际规格页集合不一致 | 比对模块清单 |
| mermaid 图陈旧/失配 | `overview.md` 的依赖图/流转图与"功能模块全景"表或各页交叉引用不一致（漏画新模块、连边过时、状态机与 BR 不符）；或规格页该有流程图却缺失 | 比对 mermaid 节点/连边 ↔ 模块清单与交叉引用 |
| 数据缺口 | 重要概念缺乏依据，可通过补一次变更或调研补齐 | LLM 判断 |

### 3. 输出体检报告
向用户呈现结构化清单（不写入文件，除非用户要求）：
```markdown
# llmwiki 体检报告 — <YYYY-MM-DD>

## 严重（建议尽快处理）
- [矛盾] <模块A-spec.md> 与 <模块B-spec.md> 对 <行为> 描述冲突 → 建议：<...>

## 一般
- [孤立页] <模块C-spec.md> 无任何交叉引用 → 建议：<...>
- [索引失配] index.md 缺少 <模块D> 条目 → 建议：补录

## 提示（可选优化）
- [缺概念页] "<术语>" 在 3 个规格中出现，建议建立 glossary.md 条目
- [数据缺口] <...> 可通过调研/新变更补齐

## 建议探索的新问题
- <LLM 提出的、值得进一步调研或发起变更的方向>
```

### 4. 修复（逐项确认）
对可自动修复的项（补交叉引用、同步 index.md 摘要、修正 frontmatter、更新 overview 表格），逐项征求用户同意后执行。
涉及规格内容判断的项（矛盾、陈旧声明）只给修复方案，由用户决定是否新发起一个变更来处理——**不在体检中直接改写规格事实**。

### 5. 记录体检
向 `llmwiki/log.md` 追加一条：
```
## [<YYYY-MM-DD>] lint | 体检
检出 X 项（严重 a / 一般 b / 提示 c），已修复 <清单>，遗留 <清单>。
```

## 完成标志
体检报告已呈现，用户确认的修复已落地，`log.md` 记录一条 `lint`。
